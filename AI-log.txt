SESSION 6 - RateLimitService (Fixed Window)

PROMPT:
  Create RateLimitService using Fixed Window algorithm. Methods:
  IsRateLimitedAsync, LogRequestAsync, GetRemainingRequestsAsync,
  GetResetTimeInSecondsAsync. Use RequestLogs table with Timestamp.

OUTPUT RECEIVED:
  Service counting requests with DateTime.UtcNow.AddMinutes(-1) window,
  comparing to user.RateLimit. GetResetTimeInSecondsAsync finds the
  oldest request in the window and calculates remaining seconds.

HOW USED:
  Used exactly as provided. This is the core rate limiting logic.

DISCUSSION:
  Asked Claude about the difference between Fixed Window, Sliding Window,
  Token Bucket, and Leaky Bucket algorithms. Claude explained:
  - Fixed Window: simple, low memory, has boundary burst problem
  - Token Bucket: gradual refill, used by AWS/Stripe, more user-friendly
  - Sliding Window: most accurate, higher memory usage
  Decision: kept Fixed Window for simplicity, noted Token Bucket as the
  recommended production alternative in the presentation.

SESSION 9 - StripeService

PROMPT:
  Create StripeService with CreateCheckoutSessionAsync (create/reuse
  Stripe Customer, subscription mode, metadata with userId and planId),
  CancelSubscriptionAsync (CancelAtPeriodEnd=true, do NOT change plan
  immediately), HandleWebhookAsync (checkout.session.completed,
  customer.subscription.deleted, customer.subscription.updated).

OUTPUT RECEIVED:
  Complete StripeService. CancelSubscriptionAsync uses
  SubscriptionUpdateOptions { CancelAtPeriodEnd = true }.
  HandleCheckoutCompletedAsync updates user plan and creates/updates
  Subscription record. HandleSubscriptionDeletedAsync downgrades to Free.

ISSUES ENCOUNTERED:
  1. 'Subscription' does not contain a definition for 'CurrentPeriodEnd'
     Claude investigated and found that Stripe.net version varies.
     Solution: added StripeSubscriptionExtensions static class that
     parses StripeResponse.Content JSON to extract current_period_end
     unix timestamp, with fallback to StartDate + 1 month.

  2. CancelSubscription was immediately setting plan to Free.
     Claude identified the bug: plan change was happening synchronously
     in CancelSubscriptionAsync. Fixed by removing the plan change
     from cancel method and only applying it in
     HandleSubscriptionDeletedAsync (triggered by Stripe webhook
     after the billing period ends).

HOW USED:
  Used with modifications described above.

SESSION 10 - RateLimitMiddleware

PROMPT:
  Create RateLimitMiddleware that: skips auth/swagger/hubs/status/risk-score
  paths, authenticates via X-API-Key OR JWT, returns JSON 401/429 errors,
  adds headers via OnStarting callback, logs after _next(), sends SignalR
  StatusUpdate and RiskScoreUpdate to user group after each request.
  Inject IHubContext<DashboardHub> in constructor.

OUTPUT RECEIVED:
  Complete middleware. Used OnStarting for headers to prevent
  "Headers are read-only" exception. SignalR sends to group "user_{userId}".

ISSUES ENCOUNTERED:
  1. "Headers are read-only, response has already started" — trying to add
     headers after response started. Claude fixed with OnStarting callback.

  2. ERR_INCOMPLETE_CHUNKED_ENCODING — caused by async operations in
     OnStarting. Fixed by moving heavy async calls (GetRemainingRequests,
     GetRiskScore) outside OnStarting and only doing the lightweight
     TryAdd inside.

  3. Dashboard status requests consuming rate limit — /data/status was
     being counted every 1 second. Fixed by adding path exclusions for
     /data/status and /data/risk-score in the middleware skip list.

HOW USED:
  Used with all three fixes applied.

SESSION 14 - AWS Elastic Beanstalk Deployment

PROMPT:
  How do I deploy ASP.NET Core 9 to AWS Elastic Beanstalk?

OUTPUT RECEIVED:
  Step-by-step guide: publish → create Procfile → create ZIP → upload.
  Specific warning about using ZipFile.CreateFromDirectory instead of
  Compress-Archive to avoid Windows backslash path issues on Linux.

ISSUES ENCOUNTERED:
  1. "backslashes as path separators" error — ZIP was created with
     Compress-Archive which uses Windows paths. Fixed by using
     System.IO.Compression.ZipFile.CreateFromDirectory.

  2. 502 Bad Gateway — application not listening on correct port.
     Fixed by removing manual app.Urls.Add() — AWS Elastic Beanstalk
     manages port assignment automatically via PORT env variable.

  3. 404 on /swagger — Swagger was inside if(Development) block.
     Fixed by moving UseSwagger() and UseSwaggerUI() outside the
     environment check so they always activate.

  4. SQL Server connection failed — RDS Security Group did not allow
     inbound traffic on port 1433. Fixed by adding MSSQL inbound rule
     in AWS Security Group.

HOW USED:
  Used with all 4 fixes applied. Successfully deployed on attempt 5 (v5).

SESSION 17 - Stripe Webhook Configuration

PROMPT:
  How do I configure Stripe webhooks for AWS production? I used Stripe CLI
  locally. Now the API is on Elastic Beanstalk.

OUTPUT RECEIVED:
  Explanation that Stripe CLI webhook secret (whsec_...) is only valid for
  local forwarding. For production, must create a new endpoint in
  Stripe Dashboard → Developers → Webhooks pointing to the EB URL.
  New endpoint generates a different signing secret that must be set
  as STRIPE_WEBHOOK_SECRET in EB environment variables.

HOW USED:
  Created webhook endpoint in Stripe Dashboard. Updated STRIPE_WEBHOOK_SECRET
  in Elastic Beanstalk environment properties. Redeployed.

SESSION 39 - Debugging & Minor Fixes

Various small debugging sessions:

1. EF Core migration error "EnableRetryOnFailure" — SQL Server connection
   failing on startup. Added EnableRetryOnFailure(5, 30s, null) to
   DbContext configuration.

2. useEffect dependency warnings in React — multiple ESLint warnings about
   fetchStatus not in dependency array. Fixed with useCallback and proper
   dependency declarations.

3. "Cannot access variable before it is declared" for fetchStatus in
   Dashboard — function was declared after useEffect that called it.
   Fixed by moving function declaration above useEffect.

4. "Calling setState synchronously within an effect" — ESLint warning.
   Resolved by disabling the specific ESLint rule with comment or
   restructuring with useCallback.

5. Stripe CurrentPeriodEnd property not found — Stripe.net version
   variation. Fixed by writing StripeSubscriptionExtensions class that
   parses the raw StripeResponse.Content JSON.
